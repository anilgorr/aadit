# Content sections

Four content sections live alongside the hub/service architecture at `artifacts/aadit-tech/`:

- **Blog** — Velite `posts` collection. Routes: `/blog` (canonical page 1), `/blog/page/[page]` (pages 2..N only), `/blog/[slug]` (post). Helpers in `lib/blog.ts`.
- **Glossary** — static TS data (`lib/glossary.ts`), `/glossary` + `/glossary/[slug]`. Definition-first layout + `DefinedTerm` JSON-LD for featured snippets.
- **Industries** — static TS data (`lib/industries.ts`), `/industries` + `/industries/[slug]`.
- **Resources** — `/whitepapers` + `/ebook` share `components/resource-listing.tsx`, driven by `lib/resources.ts`. `RESOURCES` is intentionally empty (no real assets supplied) → designed empty state renders.

## Non-obvious constraints

- **TOC anchors depend on `rehype-slug`.** Velite `s.toc()` generates urls like `#our-focus` using github-slugger. For those to actually scroll, rendered MDX headings need matching `id`s. That only happens because `rehype-slug` is registered in `velite.config.ts` `mdx.rehypePlugins` AND the custom MDX heading components spread `{...props}` (so the compiled `id` prop flows through). Remove either and the TOC links silently break.
  **Why:** the id and the toc url are generated by two separate mechanisms that only agree because both use github-slugger.

- **Reading time / TOC come from Velite schema helpers**, not hand-rolled: `metadata: s.metadata()` (→ `{readingTime, wordCount}`) and `toc: s.toc()`. TOC is only rendered for posts with `toc.length >= 3` ("longer posts").

- **Route precedence:** the static section routes win over the root `[hub]`/`[hub]/[slug]` dynamic segments, so no collision. Keep new top-level sections as static folders.

## Company/legal pages

- `/contact`, `/about`, `/careers`, `/privacy-policy` all exist as static `app/<name>/page.tsx` pages (same Header/Section/Card/Breadcrumbs/buildMetadata pattern as service pages). Footer "Company" links wired to /about + /careers + /contact.
- Contact page holds 3 verbatim offices (India Bangalore-560094, USA Bellevue WA, Dubai FZCO) + ContactForm.
- Privacy policy carries an amber "needs legal counsel review" banner — it's a template, not vetted legal text.

## Still-open site-wide gap

- Header/footer **service mega-menu** items (cyber/compliance/IT services) still point to `#` placeholders — service routes exist but aren't linked from the menu. Left as-is intentionally across content batches (owner hasn't wired them). Wiring them is a discoverability follow-up, not a bug.
